### 根因分析：15m 墙

标定脚本使用了 `ground_clutter_db=6.0`（主生成脚本的甜点）但 **Pareto ratio=0.95 过严** + **baseline [0,0,0] 检查**。更根本的问题：

- 在 1000×1000m 区域、v_max=15m/s 约束下，SCA-FP 的最优解在物理边界外 300-500m 处
- 所有 10 次 random restart 从不同方向出发，最终全部撞上 15m/s 速度墙的同一个边界点
- 在凸局部空间里，从不同初始点出发的梯度下降全部收敛到同一硬约束边界
- **对同一局部解施加 ε 扰动后重跑 SCA-FP，所有 ε 值的回弹步数都是 3.0**（刚好是 `max_outer_iters=30` 下 3 个 ADMM 外循环）
- 方差 = 0 → snap-back 测试失去区分度 → 无法优选 Chosen

### 影响

1. ε 标定失败 — 无法选出"最优 ε"
2. snap-back 测试成为算力浪费 — 每个 env 跑 3 次 SCA-FP 但没有区分度
3. baseline [0,0,0] warm_start 同样收敛到同一 15m 边界点 → `s.utility > baseline_util` 总是 false → 80% 环境被误杀

## 2. Bug 狩猎 — 5 个修复

### Bug 13: 负数 utility 的 Pareto 阈值灾难

**位置**: `oracle_generator.py:_pareto_filter`, `calibrate_epsilon.py:_pareto_filter`

**症状**: 39/50 环境被 Pareto filter 跳过 — 包括 `max_utility = -10` 的最优解被自己踢掉。

**根因**:
```python
# 修复前:
threshold = max_util * 0.85  # max_util = -10 → threshold = -8.5
# 检查: -10 >= -8.5? → False! 最优解被丢弃
```

当 utility 为负数时，`max_util * ratio` 的绝对值小于 `max_util`，导致阈值比最优解还高。所有解都无法通过过滤。

**修复** (`f9c968b`):
```python
# 用绝对值偏移: 兼容正负号
threshold = max_util - abs(max_util) * (1.0 - ratio)
# max_util = -10: threshold = -10 - 10 × 0.15 = -11.5 ✓
# max_util = +10: threshold = 10  - 10 × 0.15 = 8.5   ✓
```

**教训**: 物理效用函数可能为负。不等式推导必须验证符号。

### Bug 14: `delta_q_perturbed` 未定义 (NameError)

**位置**: `oracle_generator.py:_run_snapback_test` 第 298 行

**症状**: `delta_q_perturbed = perturbed_q - q_current` 这一行缺失 — 变量使用但从未赋值。当 `_run_snapback_test` 被调用时会抛 `NameError`。

**根因**: Grilling 终稿代码落地时的遗漏。变量在 clamp 块和 warm_start 构造之间被使用，但赋值语句被遗漏。

**修复** (`f9c968b`):
```python
# clamp 块之后, warm_start 构造之前:
delta_q_perturbed = perturbed_q - q_current
```

**教训**: 未调用的死代码路径也需要完整 — 应被 lint 或类型检查捕获。

### Bug 15: Baseline 检查误杀 80% 环境

**位置**: `oracle_generator.py:_pareto_filter`

**症状**: 80% 环境在 Pareto filter 后被丢弃。

**根因**: 15m 墙场景中，`[0,0,0]` warm_start 的 SCA-FP 与 best-of-N 收敛到相同的边界点。`s.utility > baseline_util` 始终为 False → 被误杀。

**修复** (`4fc398d`): 从 `_pareto_filter` 完全移除 `baseline_util` 参数和检查。utility_ratio 门（默认 0.85）足以过滤劣质解。

**教训**: 在硬约束边界占主导的场景中，baseline 与最优解不可区分 → baseline 检查失去意义。

### Bug 16: DPO 对退化 — Chosen ≈ Rejected

**位置**: `oracle_generator.py:_construct_rejected`

**症状**: DPO 的 Chosen 和 Rejected 的 δ_q 近乎相同（`np.allclose` with atol=0.5 pass），导致偏好信号为零。

**根因**: 15m 墙下所有 3 次 SCA-FP restart 收敛到同一边界点。取 "worst valid" SCA-FP 解作为 Rejected 时，其 Q 与 Chosen（best）几乎相同。

**修复** (`f9c968b`): 在 `_construct_rejected` 开头添加退化检测：
```python
if np.allclose(worst_valid.Q, solutions[0].Q, atol=0.5):
    # 15m 墙: 所有重启收敛到同一点 → 使用启发式物理陷阱
    delta_q = self._construct_heuristic_rejected(env_dict, q_current, rng)
    return delta_q, None  # 启发式陷阱 utility 无解析值
```

启发式陷阱有三类:
1. **Zero-move**: `delta_q = q_current` (原地不动)
2. **Greedy-line**: 各 UAV 直飞最近用户 (忽略 sensing/load balance)
3. **Ghost-direction**: `delta_q = -chosen_delta_q` (反向逃逸)

**教训**: 当 solver 的最优解空间崩塌为单点时，真正的偏好信号来自物理启发式陷阱而非 solver restarts。

### Bug 17: Snap-back 测试无区分度 → 算力浪费

**位置**: `oracle_generator.py:_process_one_environment`

**症状**: 所有 ε 的 snap-back 回弹步数均为 3.0，方差为 0。snap-back 测试不再能区分盆地宽度。

**根因**: 见上文"15m 墙"分析。所有候选解在相同的硬约束边界上，ε 扰动后 SCA-FP 总是在 3 个 ADMM 外循环内回到同一点。

**修复** (`4fc398d`): 从 `_process_one_environment` 完全移除 snap-back 测试。直接选 `candidates[0]`（utility 最高）作为 Chosen。节省 3 次 SCA-FP/env。

**教训**: 验证测试的信号噪声比不亚于验证系统本身。当 SNR=0 时，测试不仅无用且浪费算力。

## 3. 算力优化历程

| 阶段 | SCA-FP/env | 调用明细 | 20K env 估算 |
|------|-----------|---------|-------------|
| Grilling 终稿 | 14 | 10 restarts + 1 baseline + 3 snap-back | ~16.7h |
| 移除 baseline | 10 | 10 restarts + 移除 baseline (1→0) | ~11.9h |
| 移除 snapback | 10 | 10 restarts | ~11.9h |
| **减少 restarts** | **3** | **3 restarts (70% 算力节省)** | **~4h** |

**3 restarts 的理由**: 15m 墙场景中，所有 restart 收敛到同一点。保留 3 次是工程冗余 — 防止极个别环境因矩阵奇异或数值震荡导致某次求解崩溃。**用 3 次而非 1 次，是算力与鲁棒性的精准平衡。**

**实际**: 3.98h, 0.7s/env, 100% yield。

## 9. 关键经验

1. **在硬约束边界占主导时，解决方案空间的多样性来自约束本身而非初始点。** extra restarts 不会产生不同的最优解 — 只会让 10 个人在同一个滑梯上滑下去，落在同一个坑里。

2. **测试的信号噪声比是测试有效性的前提。** snap-back 测试在 SNR=0 时不仅无用，还消耗 21% 的算力 (3/14)。先验证测试是否有区分度，再全量执行。

3. **负数物理量是合理的。** 任何假定 "max > 0" 的乘法阈值公式在负数域会反转不等式方向。用 `max - abs(max) × (1 - ratio)` 统一正负号。

4. **物理启发式陷阱的数据价值可能超过 solver restarts。** 当最优解空间崩塌时，精心设计的物理陷阱（zero-move, greedy-line, ghost-direction）提供的偏好信号比 solver 的 worst-of-N 更强。

5. **算力优化的第一步是确认哪部分工作在产出信号。** 14→3 SCA-FP/env 节省 79% 算力，数据质量零损失。不是因为"砍得够狠"，而是因为确认了被砍掉的 11 次调用确实不产出任何增量信息。
