## 三、为什么停在 step 200

SFT 训练到 step 200 时做了 checkpoint 评估，发现：

1. **投影头工作正常** — `delta_q` 对 `q_current` 变化有响应（zero-out 时 L2 ratio = 4.88×）
2. **文本生成塌缩** — 模型自回归生成几乎相同的 JSON，不管输入什么环境
3. **训练尚未收敛** — step 200 的 projection head delta 值与 ground truth 还有差距

**决策**: 先评估 checkpoint 质量，再决定是继续 SFT 还是直接进 DPO。不盲目跑满 3 epochs。

## 四、架构速览

```
Prompt (环境文本描述)
    │
    ▼
Gemma 3 12B (frozen, LoRA r=16 α=32)
    │
    ├──→ lm_head → 自回归文本生成 (JSON 响应)
    │
    └──→ Control Token hidden states (8×3840)
            │
            ▼
        ConstraintProjectionHead (3 heads: Q/A/P)
            │
            ▼
        warmstart = {delta_q (4×3), delta_a (4×20), delta_p (4×21)}
            │
            ▼
        SCA-FP Solver (迭代优化, 热启动加速)
```

**两种使用模式**:
- `forward()` — 训练：同时输出 logits + control_states
- `generate_warmstart()` — 推理：单次前向，不生成文本，只输出 delta 预测

**关键参数**:
| 参数 | 值 |
|---|---|
| LoRA rank | 16, alpha=32 |
| Control tokens | 8 个特殊 token |
| Projection head | 2-layer MLP [256, 256] |
| seq_length | 3456 |
| 精度 | bf16 全精度（96GB 无需量化） |
| 有效 batch | 16 (SFT: bs=2 × grad_accum=8) |
| 损失 | L_SFT + 0.5 × L_ctl |
| 优化器 | AdamW, 投影头 lr=1e-3, LoRA lr=2e-4 |

## 五、Checkpoint 评估结果

### 5.1 投影头 (generate_warmstart) — ✅ 工作

这是核心功能，直接决定 warmstart 有没有用：

- **delta_q**: 对 q_current 敏感。q_current 归零时 delta_q 响应比 4.88×
- **delta_q 数值**: mean≈-4.2, std≈6.4, range=[-13.4, 5.3] — 合理
- **delta_a**: mean=0.25, 不随 q_current 变化（可能是架构设计如此）
- **delta_p**: mean=0.05, 不随 q_current 变化（同上）
- **delta_q 对小幅扰动不敏感**: ±10m 随机偏移时 L2 ratio = 0.0000 — 这是好事（鲁棒性）

### 5.2 文本生成 — ❌ 模板塌缩

5 个不同环境生成的 JSON **几乎完全相同**：
- 所有 `delta_q` 都是 `[0, 0, Z]` 形式（Z∈{0,1,5}）
- 所有 `delta_a` 都是 one-hot 或均匀分布
- 输出在 299 chars 截断（即使 max_new_tokens=1024）

**根因**: SFT CE loss=0.005 是 teacher forcing overfitting——模型学会了"复制训练集的 JSON"，但没有学会根据输入变化输出不同的 JSON。这是典型的 language modeling on structured data 问题。

**影响评估**: 这个塌缩**可能不致命**，因为：
1. 生产路径是 `generate_warmstart()`（投影头），不走自回归文本生成
2. 文本生成在 warmstart 场景中不需要——warmstart 是单次前向
3. DPO 阶段可能修复部分文本质量问题

### 5.3 SCA-FP 加速 (Part 3) — ⚠️ 未测量

eval 脚本的 Part 3（warmstart vs cold start SCA-FP 迭代次数对比）因为 `.numpy()` bug 未能运行。**这是判断 checkpoint 能不能用的最关键指标**。

**已在 commit `fb7332e` 修复**，只需在服务器上重跑。

## 八、关键经验教训

### 绝对不要做的事

1. **不要 `import unsloth`** — 全局 monkey-patch transformers，与 SDPA + gradient checkpointing 冲突，导致 CheckpointError
2. **不要往系统盘写数据** — `/root/` 只有 30GB，所有输出用 `/root/autodl-tmp/`
3. **不要用 flash_attention_2** — Blackwell sm_120 没有预编译 FA2 wheel
4. **不要 deepcopy DPO reference model** — 会 OOM（48GB×2），用独立加载
5. **不要在 bs=2 跑 DPO** — 双模型 + 4×logits 会爆 96GB

### 做对了的事

1. **纯 PyTorch CE** — 比 Unsloth 慢但稳定，不引入隐藏 bug
2. **SDPA attention** — cuDNN Fused Attention 自动选择最优内核
3. **bf16 全精度** — 96GB VRAM 足够，不需要量化的精度损失
4. **expandable_segments** — 动态释放 CUDA 缓存，减少碎片化 OOM
5. **overfitting test** — 5 样本快速验证训练回路，避免全量训练才发现 bug
